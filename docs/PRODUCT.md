# Product

Use this document to describe what the project is for before implementation
details harden around the wrong assumptions.

## Status

Product direction is not defined yet. Until this is replaced with real content,
agents should treat the repo as a generic software starter and avoid making
irreversible product decisions.

## To Define

### Problem

- What problem does this project solve?
- Why is it worth solving now?

### Users

- Who is the primary user?
- Who is explicitly not the target user?

### Core Outcome

- What should a successful user be able to do?
- What is the smallest version of that outcome?

### Non-Goals

- What should this project not try to do?
- What adjacent ideas are intentionally out of scope?

### Constraints

- Are there platform, budget, legal, or timeline constraints?
- Are there preferred languages or deployment targets?

## Guidance For Agents

Until this document is filled in:

- do not invent product requirements
- prefer infrastructure and documentation improvements over feature work
- ask for clarification before building user-facing behavior
